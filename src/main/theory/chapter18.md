# 함수형 관점으로 생각하기
## 선언형 프로그래밍
> 함수형 프로그래밍을 보기 전, 먼저 명령형과 선언형을 비교해보자.

* 명령형 : '어떻게'에 집중한다.
* 선언형 : '무엇을'에 집중한다.

명령형 예시  
```text
Transaction mostExpensive = transaction.get(0);
if (mostExpensive == null)
    throw new IllegalArgumentException("Empty list of Transactions");
    
for (Transaction t: transactions.subList(1, transactions.size())) {
    if (t.getValue() > mostExpensive.getValue()) {
        mostExpensive = t;
    }
}
```

선언형 예시  
```text
Optional<Transaction> mostExpensive = 
    transactions.stream()
                .max(comparing(Transaction::getValue));
```

## 함수형 프로그래밍이란?
* 우선, 함수형 프로그래밍에서의 함수란?
  * 0개 이상의 인수를 가진다
  * 1개 이상의 결과를 반환하나 부작용이 없다
* 부작용이 없다?
  * 입력이 같으면 항상 출력이 같아야 한다
  * 외부 상태를 변경하지 않는다

1️⃣프로그래밍에서 부작용이란:
> 🔥 함수가 자기 바깥의 상태를 변경하는 것

예를 들면 이런 것들이다.
```text
int total = 0;

void add(int value) {
    total += value;  // 🔥 외부 상태 변경
}
```
여기서 add()는 값을 계산하는 게 아니라
👉 외부 변수 total을 바꿔버린다.

이게 부작용이다.

대표적인 부작용들 
* 전역 변수 변경 
* 객체 내부 상태 변경 (setter 호출 등)
* 파일에 쓰기 
* DB에 insert 
* 콘솔 출력 (System.out.println)
* 예외 던지기 
* 네트워크 호출

즉, “계산 결과” 말고 다른 것을 건드리면 부작용 ❗

2️⃣ 그럼 함수형 프로그래밍이 말하는 “부작용 없는 계산”은?

함수형 프로그래밍에서는 함수가 이런 특성을 가져야 한다고 본다:  
🎯 입력이 같으면 항상 출력이 같아야 한다.  
🎯 외부 상태를 변경하지 않는다.  

이걸 순수 함수 (pure function) 라고 한다.

ex. 순수 함수
```text
int add(int a, int b) {
    return a + b;
}
```

ex. 순수하지 않은 함수
```text
int total = 0;

int add(int value) {
    total += value;
    return total;
}
```

이 함수는:
* 호출할 때마다 결과 달라짐 
* 외부 상태에 의존 
* 테스트 어려움 
* 멀티스레드에서 위험

❌ 함수형 프로그래밍이 싫어하는 형태

### 함수형 프로그래밍의 이점
1️⃣ 예측 가능성 (Predictability)
* 언제 호출해도 동일	
* 상태에 의존하지 않음	
* 추론 가능

👉 프로그램을 “수학적으로 reasoning” 할 수 있게 됨.

2️⃣ 병렬성과 동시성에 강함

* 공유 상태 없음 
* 데이터가 불변
* 계산이 독립적

👉병렬화가 매우 쉬움.  
👉따로 뮤택스라는 걸 만들 필요가 없음.   

그래서 함수형 언어/프레임워크가 분산/동시성에 강하다.
* Scala (Akka)
* Elixir (Erlang VM)
* Haskell 
* Clojure

3️⃣ 코드 조합성 (Composability)

함수형은 작은 함수들을 조합한다.

이 구조는 작은 단위 / 독립적 / 재사용 가능.

이게 왜 강력하냐면, 복잡한 시스템을 작은 순수 조각으로 나눌 수 있기 때문.

* 특히 데이터 파이프라인에서 강력하다.
* Spark, MapReduce, Stream API, Rx 등이 전부 함수형 사고에 기반한다.

4️⃣ 테스트 용이성
순수 함수는..
* mock 필요 없음 
* DB 필요 없음 
* 환경 의존 없음

그러나, 장점만 있는 것은 아니다.
1. 초반 러닝 커브 높음 
2. 디버깅이 추상적이고 지나치면 가독성 떨어짐
3. 외부 상태를 꼭 가지고 와야 하는 경우들이 있다. (ex. I/O)

그래서 OOP + FP 혼합 모델이 인기가 많다.

정리해보면, 함수형 프로그래밍의 이점은 다음과 같다.

> 복잡성을 줄이고, 예측 가능성을 높이고, 병렬 시대에 맞는 모델을 제공하는 것

## 자바에서의 함수형 프로그래밍
> 결론부터 말하자면, 불가능하다.

그러나, 함수형 프로그래밍 스타일을 얹을 수는 있다.  
1. 지역 변수만을 변경해야 한다.
2. 메서드가 예외를 일으키면 안된다. -> Optional<T>을 쓸 수 있다.
3. 비함수형 동작을 감출 수 있는 상황에서만 부작용을 포함하는 라이브러리 함수를 사용한다.

### 참조 투명성
* 같은 인수로 함수를 호출했을 때 항상 같은 결과를 반환한다면, 이를 참조적으로 투명한 함수라고 한다.
* 근데 자바는 이게 잘 안될 때가 있다.
```text
List<Integer> makeList() {
    return new ArrayList<>(List.of(1, 2, 3));
}

List<Integer> a = makeList();
List<Integer> b = makeList();
```
* 겉보기에는 항상 [1, 2, 3]을 반환하니까 참조 투명성을 지닌 메소드가 아닌가?
  * 정답 : 그렇지 않다.
  * a와 b는 다른 메모리 주소를 가리킨다.
* 그럼 참조 투명성이 깨진건가? 원칙상은 그렇다.
* 그러나 자바에서는 '값'만 같으면 참조적으로 투명하다고 간주할 수 있다.
* 일종의 예외.

## 객체지향 vs 함수형
* 함수형 예시 1 : List<Integer>가 주어졌다고 해보자.
* 이것의 모든 부분집합으로 구성된 List<List<Integer>>를 만드는 프로그램을 만들자.
* 코드는 [이곳](../../main/java/part18/example1.java)을 참조.

이 코드는 왜 함수형인가?
✔ 입력 변경(`List<Integer> list`) 없음
✔ 전역 상태 없음. 모든 계산은 입력값과 지역변수로 이뤄짐
✔ 같은 입력 → 같은 출력
✔ 값을 반환하며 계산 누적

## 재귀와 반복
* 순수 함수형 언어에서는 반복문이 없음.
  * 왜? 반복문이 입력값의 변화를 만들 수 있기 때문
  * 그래서 재귀를 반복문 대신 이용한다.
* 그럼 재귀를 반복 대신 써야 할까?
* 자바 8 이상부터 있는 스트림이 이를 대신할 수 있다.

**근데, 무조건 재귀가 좋을까?**
-> 그렇지 않다. 재귀를 잘못 쓰면 메모리의 스택 부분이 꽉 차게 된다.

-> StackOverFlowError가 발생할 수 있음.
